# 用户登录&授权LLM审核 Demo 说明文档
## 项目简介
本项目为**用户登录/注册+昵称违规检测+资源权限控制**演示 Demo，实现账号注册登录、基于智谱 GLM-4-Flash 大模型的社区昵称违规识别、差异化资源访问权限管控，已部署至 Render 提供公网访问。

公网访问地址：https://authdemo-am2r.onrender.com/static/index.html

## 1. 实现方式以及时间规划
### 1.1 实现方式
采用**前端静态页面 + Python Flask 轻量化后端**架构，使用本地 JSON 文件持久化用户数据，结合「本地关键词库 + 智谱 GLM-4-Flash 大模型」双层机制完成注册昵称违规检测；基于用户权限字段区分资源访问权限，资源A对所有登录用户开放，资源B仅指定授权账号可访问。最终项目部署在 Render 平台，实现公网可访问。

### 1.2 时间规划

1. 需求梳理、技术选型、整体架构与功能设计；（30分钟）
2. 前后端基础代码开发，完成注册、登录、权限校验核心功能；（30分钟）
3. 功能优化，对接LLM接口、配置超时与缓存、补充本地违规词库、调试风控逻辑；（1小时）
4. 项目适配线上环境、代码修改、打包部署与全流程联调测试。（30分钟）

## 2. 整体架构、技术栈与设计出发点
### 2.1 技术栈
- 前端：原生 HTML + JavaScript + Bootstrap
- 后端：Python + Flask + Flask-CORS
- 数据存储：本地 JSON 文件
- AI 能力：智谱 GLM-4-Flash API（昵称违规检测）
- 部署平台：Render（免费云平台）
- 依赖工具：`requests`（接口请求）、`gunicorn`（线上服务启动）

### 2.2 整体架构
采用三层极简分层架构，职责清晰、低耦合：
1. **前端交互层**：包含登录页、注册页、资源A、资源B四个页面，负责用户操作、请求发起与结果展示；
2. **后端接口层**：提供注册、登录、权限校验三大接口，处理业务逻辑、数据读写与LLM接口转发；
3. **数据&AI能力层**：JSON 文件存储账号与权限数据，结合本地违规词库 + GLM-4-Flash 大模型实现昵称分层风控。

### 2.3 设计出发点
1. 选用轻量化技术栈，降低环境配置、开发与部署成本，聚焦**登录授权、LLM 风控**两大核心需求；
2. 采用「本地规则 + 大模型」双层校验，兼顾接口响应速度与隐晦违规昵称识别能力；
3. 权限逻辑明确拆分，严格区分公开资源A与受限资源B，满足题目授权规则；
4. 增加接口超时、内存缓存、异常兜底机制，提升服务稳定性与体验；
5. 适配免费云平台部署，保证项目可公网访问，满足交付要求。

- 其次为前端页面、接口请求逻辑、批量测试脚本编写；
- 问题排查、部署排错类对话 Token 消耗最少。

## 4. 个人耗时最多的部分（不含模型调用）
1. **本地前端环境适配问题**：使用 VS Code Live Server 运行页面时，出现注册成功无法跳转、接口请求报错等问题，反复排查后确认是本地静态服务与 Flask 后端运行环境不匹配导致，切换 `python -m http.server` 可正常运行，耗费较多时间定位根因；
2. **线上部署适配改造**：前端代码硬编码本地 `127.0.0.1:5000` 地址，部署到 Render 后接口请求被拒绝，统一修改为通用相对路径、适配线上同源环境，同时排查页面404、服务休眠等部署问题；
3. **全场景测试与优化验证**：编写批量测试脚本，分类测试正常昵称、显性违规、谐音变体违规、缓存、超时等场景，核对日志与接口耗时，验证各项优化效果。

---

## 5. 该场景下优先级最高的部分
结合**账号授权 + LLM 昵称风控**的业务场景，优先级从高到低排序：
1. **账号注册、登录基础功能**：是整个系统的底座，必须保证稳定可用；
2. **资源差异化权限控制**：区分资源A、资源B的访问权限，是本次 Demo 核心业务要求；
3. **LLM 昵称违规检测能力**：题目明确要求引入大模型完成风控，为核心考核点；
4. **服务稳定性与性能优化**：接口超时、缓存、本地词库等优化，保障使用体验；
5. **页面样式、交互美化**：仅做视觉优化，优先级最低。
